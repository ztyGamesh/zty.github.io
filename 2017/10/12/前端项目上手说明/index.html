公司的CMS项目中使用了webpack、redux、react、es6技术栈，总结了一套配置方案，以便下此做项目时可以快速开始， 并进行持续优化。

项目结构规划

每个模块相关的css、img、js文件都放在一起，比较直观，删除模块时也会方便许多。

测试文件也同样放在一起(还未添加)，这样哪些模块有没有写测试，哪些测试应该一起随模块删除，一目了然。

上手开发流程

开发人员角色:

level: 0 可以增加新的路由

  1. mock –> allMenu.js，增加新的目录与路由
  2. layouts –> content.js –> 增加新的路由&与之相对应一级目录 例如叫demo
  3. views –> 创建新的一级目录 –> 创建新的二级目录 目录名为demo 同时创建index.js 这就是页面组件的入口
  4. components –> 创建对应二级目录demo的各种组件 并同时在view中创建的index.js中引入创建的组件

level: 1 只专注于开发页面的功能

  1. views –> 找到1级目录(大的分类) –> 找到2级目录(页面级别目录) –> index.js 页面组件入口
  2. components –> 创建对应页面的组件集合目录 –> 开发针对于页面相对应的组件 –> 在上一步中的index.js中引入

项目的一些说明

component的组织

目前组件是按照页面维度划分的，并没有按照理想的开发流程，将组件分为

  • pure 纯组件
  • pageComponent 实际使用的组件

    原因:
  • 没有明确的需求，给出要设计适合我们项目中的组件，所以还没有设计
  • 开发时间紧张，还没有去抽象适合项目的组件 目标:
  • 抽象出适合公司业务的组件，以功能或者需求作为一级目录，在component目录下划分纯组件
  • container目录中以页面维度作为一级目录，包装纯组件，生成pageComonent组件。供页面使用

action的放置

目前所有针对redux🌲的action。都是放在相应的reducer.js文件中

这样的好处是:

  1. 传统的做法是将 action 单独切出去一个目录actions，让action和reducer分开。
  2. 这种做法对于我们目前的项目来说可能有点过度优化了，其实多数情况下特定的 action 只会影响到特定的 reducer。 而分开会给我们的维护和理解代码带来额外的不必要的负担，有种矫枉过正的意味。但是这里没有放之四海皆准的规则。
  3. 所以actions目录我也留着， 大家可以多参考、多尝试，找到适合自己开发需求的方案。
  4. 目前将action放到reducers里面还是比较方便的，具体可以到项目中了解
