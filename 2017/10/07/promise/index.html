Promise是一种js的异步编程解决方案，它比传统的解决方案–回调函数和事件–更加强大。

基本概念

单线程

javascript是单线程的。 所谓单线程，是指JS引擎负责解释和执行js代码的线程只有一个。 在本文当中，我们称它为主线程

但是实际上还可能存在其他的线程。

例如:

  • 处理ajax请求的线程
  • 处理DOM事件的线程
  • 定时器线程
  • 读写文件的线程(NodeJS)
  • 等等

这些线程可能存在于JS引擎之内，也可能存在于JS引擎之外。

本文当中，统称它们为工作线程

同步&异步

来看看对于同步函数和异步函数的定义

具体来说，对于函数A

A(args...);
  • 同步

    如果在函数A返回的时候，调用者能够得到预期的结果，那么这个函数就是同步的。 ps: 所谓的预期的结果是说，拿到了函数预期的返回值or实现了预期的效果
//1 第一个函数返回的时候，在控制台看到了"ok"，实现了预期的效果
console.log('ok');
//2 第二个函数返回的时候，拿到了预期的返回值,即返回值"ok"
function test() {
  return "ok"
}
test()

console.log&test 就是所谓的同步函数；

  • 异步

    如果在函数A返回的时候，调用者还不能够得到预期的效果，而是需要在将来通过一定的手段得到，那么这个函数就是异步的。
fs.readFile('foo.txt', 'uft8', function (err, data) {
    console.log(data)
})

在上面的代码中，我们希望通过fs.readFile函数读取文件foo.txt中的内容，并打印控制台。但是在fs.readFile返回的时候，我们期望的结果并不会发生，而是要等到文件全部读取完成之后。如果文件很大可能要读取很长的时间。

前端开发中，比较常见的异步场景就是ajax请求了。通俗地说:

异步ajax:

主线程: ‘ajax线程你好，请你帮我发送一个HTTP请求，请求的相关参数我都给你了’

ajax线程: ‘好的，主线程，我立刻发送HTTP请求，但是可能会花点时间，你先去忙别的吧，拿到响应后我会告诉你’

主线程: ‘好的，拿到响应了告诉我一声’

同步ajax:

主线程: ‘ajax线程你好，请你帮我发送一个HTTP请求，请求的相关参数我都给你了’

ajax线程: ‘…’

主线程: ‘ajax线程，你这边是什么情况，怎么不说话’

ajax线程: ‘…’

主线程: ‘ajax线程，我在等你回复呢，你什么情况?’

ajax线程: ‘…’

(一段时间后)

主线程: ‘ajax线程，求求你说句话吧’

ajax线程: ‘主线程，不好意思，我工作的时候不能说话，你的请求已经发送完了，这里是响应数据，给你’

可以看出来，正是由于js是单线程的，同步对于耗时的操作或者不确定时间的操作会实现主线程的阻塞，而异步可以避免主线程的阻塞。 所以对于耗时操作或者不确定时间的操作，使用异步函数处理就成了理所当然的选择。

于是，异步编程出现了，它出现的原因如上所述。

异步过程的构成

从上面的描述可以看出，异步函数 的调用实际上很快就完成了。 但是后面还有:

  • 工作线程执行异步任务
  • 工作线程通知主线程
  • 主线程调用回调函数
  • 等等

    我们把整个过程叫做 异步过程

总结一下:

主线程发起一个异步请求，相应的工作线程收到请求并告知主线程已收到(异步函数返回);主线程可以继续执行后面的代码，同时工作线程执行异步任务； 工作线程完成工作后，通知主线程；主线程收到通知后，执行一定的动作(回调函数)

异步函数通常具有如下的形式:

A(args..., callbackFn)

它可以叫做异步任务的发起函数，或者叫做异步任务的注册函数。

所以，从主线程的角度来看，一个异步任务包括下面两个因素:

  • 发起函数A
  • 回调函数callbackFn

它们都是在主线程上调用的，其中注册函数用来发起异步过程，回调函数用来异步任务返回的结果。

举个例子:

setTimeout(fn, 1000);

其中setTimeout就是异步过程的发起函数，fn就是异步过程的回调函数。

tip:A(args..., fn)只是一种抽象的表示，并不代表回调函数一定要作为发起函数的参数

var xhr = new XMLHttpRequest();
xhr.onreadystatechange = fn; //添加回调函数  
xhr.open('GET', url);
xhr.send() //发起函数

发起函数和回调函数就是分离的。

消息队列和事件循环

前面说到，异步过程中，工作线程在异步操作完成后需要通知主线程。那么这个 通知机制 是怎么实现的呢?

答案是: 利用 消息队列和事件循环。

用一句话概括:

工作线程将消息放到消息队列，主线程通过事件循环过程去取消息。

  • 消息队列: 消息队列是一个先进先出的队列，里面存放着各种消息
  • 事件循环: 事件循环是指主线程重复从消息队列中取消息、执行的过程

实际上，主线程只会做一件事情，就是从消息队列里面取消息、执行消息，再取消息、再执行消息。当消息队列为空时，就会等待直到消息队列变成非空。

而且主线程只有在将当前的消息执行完以后，才会去取下一个消息。

这种机制就叫做事件循环机制，取一个消息并执行的过程叫一次循环。

事件循环用代码表示:

while(true) {
  var message = queue.get()
  execute(message)
}

那么，消息队列中放的 消息 具体是什么东西呢？ 消息的结构当然跟具体的实现有关，但是为了简单起见，我们可以认为:

消息就是注册异步函数时候添加的回调函数

再次以异步ajax为例:

$.ajax(url, function (res) {
    console.log(res)
})

主线程在发起ajax请求后，会继续执行其他代码。 ajax线程负责请求url，拿到响应后，它会把响应封装成一个js对象，然后构造一条消息:

// 消息队列中的消息就长这个样子  
var message = function () {
  callbackFn(res)
}

其中callbackFn就是前面代码得到成功响应时的回调函数。

主线程在执行完当前循环中所有代码之后， 就会到消息队列中取出这条消息(也就是message函数),并执行它。

到此为止，就完成了工作线程对主线程的通知，回调函数也就得到了执行。如果一开始主线程没有提供回调函数，ajax线程在收到了HTTP响应后，就没有必要通知主线程，从而也没有必要往消息队列里面放消息。

从上文中我们也得到一个明显的结论:

异步过程的回调函数，一定不在当前这一轮事件循环中。

异步与事件

上文中说的”事件循环”, 为什么里面有个事件呢？那是因为:

消息队列中的每一条消息实际上都对应着一个事件。

上文中一直没有提到一个很重要的异步过程: DOM事件

var button = document.getElementById('#btn')
button.addEventListener('click', function (e) {
    console.log(e)
})

从事件的角度来看,上述代码表示: 在按钮上添加了一个鼠标单机事件的事件监听器；但用户点击按钮时，鼠标单机事件触发，事件监听器函数被调用。

从异步的角度来看，addEventListener函数就是就是异步过程的发起函数，事件监听器函数就是异步过程的回调函数。事件触发时，表示异步任务完成，会将事件监听函数封装成一条消息放到消息队列中，等待主线程执行。

事件的概念其实上并不是必须的，事件机制实际上就是异步过程的通知机制。

另一方面，所有的异步过程也都是可以用事件来描述的。 例如: setTimeout可以看成对应一个 ‘时间节点’的事件。

setTimeout(fn, 1000)
---  可以看成  
timer.addEventListener('timeout', 1000, fn);

生产者与消费者

异步过程:

工作线程是生产者，主线程是消费者(只有一个消费者).工作线程执行异步任务，执行完成后把对应的回调函数封装成一条消息放到消息队列中; 当主线程中的任务执行完以后，主线程会不断地从消息队列中取出消息并执行，直到消息队列为空。
